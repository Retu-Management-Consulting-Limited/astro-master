# 设计 Spec：Molly 解读质量评测尺（eval harness v1）

**日期**：2026-06-14
**项目**：astro-master（Molly / vapeincity.com）
**状态**：设计定稿 + 2026-06-15 重审（对齐已存在的 CI/PR 现实）· 待写实现计划
**作者**：Claude（与 Kevin 头脑风暴）
**定位**：自我优化平台的 **Phase 1 地基**（见 `2026-06-14-self-optimization-strategy-design.md` §6）——「评测先行」，所有后续优化的度量衬底。
**⚡ 优先级（2026-06-15 重审上调为紧急）**：现实是一天 merge 十几个 PR、多 agent 并行、大量改解读相关文案，而现有 CI（`.github/workflows/ci.yml`）只挡功能/类型/构建回归，**没有任何东西挡「解读变差/跑调/更套话」**。这把尺子是当下快速迭代缺的那道质量棘轮，不是从容的未来地基。

---

## 1. 目标（一句话）

给 Molly 的解读造一把可量化的「质量尺子」：给定真测试者的盘 → 跑出 AI 解读 → 多维打分 → **改 prompt/模型时立刻知道变好还是变坏（且没退步）**。没有这把尺子，任何「让 Molly 越说越准」的改动都是凭感觉。

---

## 2. 锁定的决定（头脑风暴结论）

| 决策点 | 决定 |
|---|---|
| 裁判 | **混合**：LLM 裁判日常批量跑 + 定期用真人评分校准 |
| 样本 | **共鸣感用真测试者盘**（少而真，随用户长大，PII 不入 git）；**CI 安全维用合成盘**（无 PII，见下「拆开跑」） |
| 共鸣感金标准来源 | **产品内「准度评分」**（一举两得：评测金标准 + 「越用越准」校准回路数据源） |
| v1 评测范围 | **首读 + 主题深读（全维）+ 对话（轻量子集）** |
| 用途 | 双重：**回归闸**（防退步）+ **指南针**（指下一步改哪） |
| **跑法（2026-06-15 定）** | **拆开**：① 安全三维（占星正确/反套话/安全语气）→ **进 CI 当硬闸**，用合成盘、无 PII、可阻断；② 共鸣感（需真盘+真人评分）→ **单独非阻断校准任务**，受控环境跑。既有实时质量护栏，又不把真盘 PII/密钥/每-PR token 塞进 CI。 |

### 2.1 为什么这样拆（PII × CI 张力的解）
质量 eval 要真盘+API key+烧 token，与「真盘不进 CI/不外泄」冲突。解法是**按维度拆**：四维里只有「共鸣感」真需要真人盘+真人评分；另三维（占星正确/反套话/安全语气）**用合成盘就能判**，不碰 PII。
- **CI 硬闸**（每个解读相关 PR）：合成盘 × {占星正确 + 反套话(交叉套盘) + 安全 + 像 Molly}。阻断。这是给"一天十几个 PR"的实时护栏。
- **共鸣感校准任务**（定期/手动，受控环境）：真测试者盘 × 真人评分 → 校准 LLM 裁判。非阻断。
- 注：CI 硬闸引入合成盘 = 对原「只用真盘」决定的*增补*（非推翻）——真盘仍只用于共鸣感；合成盘只用于 CI 那道不能进真盘的闸。

---

## 3. 四个评测维度

| 维度 | 是什么 | 怎么判 | 首读/主题 | 对话 |
|---|---|---|---|---|
| **占星正确** | 没编星位、对得上真实盘 | 确定性校验 + LLM 核对 | ✅ 全 | ✅ |
| **共鸣感**（被看穿） | 用户*觉得*「这说的就是我」 | LLM 裁判，用真人评分校准 | ✅ 全 | —（轻/不评） |
| **反套话 / 巴纳姆** | 不是「谁都觉得准」的废话 | **交叉套盘**（见 §4） | ✅ 全 | — |
| **安全 + 像不像 Molly** | 不出危险内容/不越界 + 品牌语气 | 规则（复用 `safety.ts`）+ LLM 对照 persona | ✅ 全 | ✅（对话是危机入口，安全这维更重） |

**对话做轻**：多轮、上下文相关、最难评 → v1 只按**单条回应**评「切题 + 安全 + 像 Molly + 没编星位」，不做整段多轮会话打分。扩展留后。

---

## 4. 反巴纳姆 —— 交叉套盘（核心机关）

算命最大的失败模式＝写「谁看了都觉得准」的套话，而自动优化「共鸣感分数」恰恰会把模型训成这种**巴纳姆机**（Goodhart）。机关：

> 把给 A 盘写的解读，拿去问裁判「这段有多贴 B 盘（另一个人的盘）？」
> **specificity = 贴本人(A) − 贴别人(B/C…均值)**。贴 A ≈ 贴别人 → 是废话，扣分。

- 能自动跑、相对客观，不需真人。
- 直接堵住「优化共鸣感 → 退化成套话」这条毒路（见 strategy §3 反 Goodhart 护栏）。

---

## 5. 裁判：混合 + 校准

- **日常**：LLM-as-judge 批量给四维打分（便宜、可重复）。
- **校准**：定期拿真测试者的产品内评分当**金标准**，回归校验「LLM 裁判判得跟真人一致吗」，校 LLM 裁判的偏差。
- **诚实**：共鸣感的终极真值是本人；LLM 裁判是代理。校准把代理拉近真值，但不等于真值。

---

## 6. 样本与数据

- **样本 = 真测试者的盘**。优点：共鸣感是真的；**评测集随每个新真用户一起长大、越来越厚**。
- **缺点（诚实）**：n≈10 统计弱、易过拟合 → v1 是「**校准 + 冒烟**」尺子，不是统计意义 benchmark。
- **隐私**：真出生数据严格保管、**不进 git**、不进任何会外泄的产物（参考 feedback-loop spec §12-S 的 PII 纪律）。

---

## 7. 硬依赖：产品内「准度评分」（必须先建）

现状：产品只有自由文本 `FeedbackButton`，**没有「给这段解读打分」的入口** → 没有真人金标准。

- **要加**：读完一段解读 → 轻量评分（`准 · 不准` 或 `1-5` + 可选一句）→ 落 KV（与 `addFeedback` 同套路）。
- **一举两得**：① 评测的真人金标准；② strategy 里「准度/共鸣校准回路」(sensor #2) 的数据源。**建一次，喂两处。**
- 触及面：首读/主题/对话三个解读出口加评分件；新 KV 写入（`addRating`）；admin export 加 `ratings` 段。

---

## 8. 用途（双重，落到两个跑法）

1. **回归闸 → 接进现有 CI**（`.github/workflows/ci.yml` 已有 typecheck/vitest/build/e2e 三/四闸）：给**解读相关 PR**加一道质量 job——合成盘跑安全三维，跌破基线 → 红、不准合。这是给"一天十几个 PR"的实时质量棘轮，是「评测先行」的脊梁。
   - ⚠️ 成本：CI 每次跑 LLM 裁判要 token + `ANTHROPIC_API_KEY` 进 CI secret。控制：只在 reading/prompt 相关路径变更时触发(path filter)；合成盘集固定小批；可设为先「告警非阻断」跑稳后再转硬闸。
2. **指南针 + 共鸣感校准 → 单独任务**（受控环境，非 CI）：真盘 × 真人评分，按维度/按盘看分布 + 校准 LLM 裁判。非阻断。

---

## 9. 建造顺序（含数据累积时序）

1. **建尺子机器 + 合成盘安全三维**（占星正确/反套话/安全语气的 LLM 裁判 + 交叉套盘 runner + 基线）→ **接进现有 CI 当质量闸**（先告警、稳后阻断）。**这步不需任何真人评分，day1 即生效**，最先堵住快速迭代的质量回归。
2. **建产品内「准度评分」** → 发给那 10 个人用 → 真人评分开始累积（同时喂 sensor #2 校准回路）。
3. **评分攒够 → 共鸣感校准任务**（单独、非 CI，真盘）：校准 LLM 裁判。

> 顺序变化（2026-06-15 重审）：把「合成盘安全三维进 CI」提到第 1 步（紧急护栏、无 PII、day1 见效），产品内评分与共鸣感校准顺延。原顺序是「先评分」，现改为「先护栏」。

---

## 10. 验收标准（这把尺子怎么算建成）

1. 产品内能给一段解读打分（准/不准 + 可选一句），落 KV，admin export 可见。✅ 可演示
2. 对一张真盘的首读，尺子能输出四维分 + 总分 + 交叉套盘的 specificity。✅ 可演示
3. 交叉套盘验证：给一段**故意写成套话**的解读，specificity 明显低于一段**贴盘的具体解读**。✅ 可演示
4. 回归闸验证：人为把 prompt 改差 → 尺子分数下跌并告警；改好 → 回升。✅ 可演示
5. 校准链路（可后补）：有 ≥N 条真人评分后，能算「LLM 裁判 vs 真人」一致度。✅ 可演示
6. 三个出口（首读/主题/对话）都能被评，对话走轻量子集。✅ 可演示
7. 隐私：真出生数据不在 git、不在外泄产物里。✅ 检查项

---

## 11. 不做 / 边界（YAGNI）

- 不做大规模合成盘 benchmark（共鸣感扩展随真用户自然发生）。注：CI 硬闸用一小批固定合成盘跑安全三维（§2.1），这是为避 PII，不是建大 benchmark。
- 不做对话的多轮整段会话打分（v1 单条评）。
- 不自动改 prompt（这把尺子只**度量**；改动走人工或 feedback-loop 执行器）。
- 不冒充统计权威（n=10，明标「校准+冒烟」性质）。

---

## 12. 留给实现计划（writing-plans 收敛）

- 评分量表：`准·不准` 二元 vs `1-5` —— 二元低摩擦、易答；1-5 信息多。倾向二元 + 可选一句，实现时定。
- 尺子在哪跑：本地脚本 / CI / 复用 feedback-loop 的执行环境 —— 倾向先本地脚本（`web/scripts/eval/`），跑通再谈自动化。
- 基线怎么存：一个 `eval-baseline.json`（非真盘数据，只存分数）。
- LLM 裁判 prompt 与四维 rubric 的具体措辞（含交叉套盘的指令）。
- 校准的「一致度」用什么指标（相关性 / 命中率）。
- 真盘样本怎么安全取用（从 KV 取真测试者盘 → 本地内存跑 → 不落盘）。
