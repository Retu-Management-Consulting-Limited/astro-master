## 统领原则：算得丰富，显得克制

后台尽量多算（喂「亮点偵測」与 AI 的信号越多越好，天文历库多算几颗零成本），前台只亮 2–3 个（守盘面美学）。「算多少」与「显示多少」是两件事。

## 1. 天体与点位

| 必算 | 产品意义 |
|---|---|
| ☉☽☿♀♂♃♄♅♆♇ 十大行星 | 标配 |
| 上升 ASC / 天顶 MC（+ 下降/天底）| 四角：性格与人生舞台 |
| **凯龙 Chiron** | 核心创伤与疗愈——直接对应「精准说穿伤口再接住」的引擎 |
| **北交点 North Node** | 此生方向/课题——对应「自我/意义」痛域 |
| **莉莉丝 Lilith（黑月）** | 压抑的暗面/女性力量——贴女性受众，给 C 毒舌加料 |

**MVP 不算**（留给 power-user 模式）：宿命点、福点、小行星（Juno 等）。
> 三个额外点的取舍依据：每个都映射一个痛域（凯龙=创伤/被看见、北交=方向/意义、莉莉丝=女性暗面），不是「越多越好」。

## 2. 宫位制：Placidus（默认，可切换）

最主流、用户预期、小红书内容默认口径。**且与潜伏式校准天作之合**：Placidus 宫头吃精确出生时间，校准引擎持续逼近 → 宫头随校准越来越准 ＝「盘越用越准」的实在来源之一。架构做成可切换，之后可加 Whole Sign（对时间不确定更稳健）当 power-user 选项。

## 3. 相位：5 主相位 + 容许度绑亮点偵測

- **显示/解读**：合(0°)、冲(180°)、拱(120°)、刑(90°)、六分(60°)。次相位后台算、不主动露。
- **容许度**：主相位 6–8°、日月放宽。
- **亮点偵測按『紧密度』排序**：orb 越小（<2–3°）＝越定义这个人＝越先亮。一张盘的「准」感来自那几条最紧的相位。

## 4. 算在哪：客户端显示 + 服务端解读（混合，已定）

- **客户端**（`astronomy-engine`，MIT，纯 JS）：即时把盘画出来 → 爽、省服务器、抗病毒流量峰值、少传一点 PII。
- **服务端**：解读与自我模型的**单一真相源**（LLM key 不下发、prompt/亮点偵測是 IP、自我模型是护城河且跨设备持久化）。
- **铁律**：两端用**同一个库、同一组设置**，防止上升/宫头漂移（client 显示天蝎、server 解读天秤 = bug）。

> 澄清：排盘的「数学」是 commodity，不是 IP；真正的 IP（亮点偵測 + AI prompt + 自我模型）无论如何在服务端，与排盘算在哪无关。

## 5. 两个必处理的工程坑

- **天文历库授权**：选 `astronomy-engine`（MIT），避开 Swiss Ephemeris 的 AGPL/数据授权雷；精度对占星绰绰有余，可上 Vercel serverless。
- **历史时区/夏令时**：出生地 → 经纬度 → **出生当时**的时区与 DST 规则（DST 规则历年多变，须 IANA 历史时区库正确回推）。需一个 geocoding + 历史时区处理层。排盘最常见出错点。
